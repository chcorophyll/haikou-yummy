# 数据清洗与爬虫工具测试计划 (Crawler Test Plan)

## 1. 概述
本文档规定了 `Haikou Yummy` 项目针对原始文本列表数据向结构化 MongoDB 数据转换的测试方案。测试代码将在 `tests/scripts/` 目录下完成。主要涵盖：文本解析的准确性、地图 API（如高德）请求的健壮性、以及处理重名/错误数据时的容错表现。

## 2. 测试策略
- **隔离测试**: 在测试中通过 `pytest-mock` 拦截发往高德/腾讯地图 API 的真实 HTTP 请求，使用预设的 Mock 响应数据进行验证（避免消耗 API 的免费额度和网络延迟影响）。
- **流程测试**: 
  文本解析引擎接收 `辖区 饭店 详细地址` 等格式字符串 -> 发送 Mock API 返回经纬度和结构化地址 -> 组装 `RestaurantCreate` 数据校验能够正确生成。

## 3. 测试用例

### 3.1 文本清理与切分准确性 (Unit Test)
- **正向用例 1 (标准四段式)**: 输入 `美兰区 鸿运饭店 蓝天路20号 138xxxx`。预期正确分割出 `美兰区`, `鸿运饭店`, `蓝天路20号`, `138xxxx`。
- **正向用例 2 (缺失段落)**: 输入用户的原版数据 `美兰区 鸿运饭店` (只有辖区和店名)。预期依然能提取出店名，其余字段为空。

### 3.2 调用地图 API 并解析 (Integration Test - Mocked)
- **正向用例 (成功抓取)**: 传入 "美兰区 鸭你吃美食大院"。Mock 返回 `pois: [{location: "110.34,20.03", address: "美苑路100号", tel: "0898-1234", biz_ext: {rating: 4.5}}]`。预期系统能将其成功转化为符合 MongoDB Pydantic 的字典模型，包含对应的 Float 经纬度。
- **反向用例 (未查找到地点)**: 传入一个地图上不存在或完全写错的店名，Mock 返回 `pois: []`。预期脚本捕捉到该信息，跳过此条目的数据库入库，并能在终端/日志中输出错误警告 (如：“未找到该店面坐标，跳过：xxx”)。

### 3.3 数据模型入库校验
- 将从 3.2 得到的标准字典喂给后端的 `RestaurantCreate` Pydantic 模型。
- **验证**: 检查是否会自动补全必需的 GeoJSON Point 结构 `[110.34, 20.03]`。如果地图 API 没有返回电话或消费数据，模型能否通过默认值安全通过验证。
